# ユースケース

このページは、Codex Maintainer Kit のどのコマンドを使えばよいか迷ったときの入口です。

## 1. メンテナンス監査をする

CodexやAIエージェントに作業を頼む前に、`audit` でメンテナンス監査をします。

良い出力: スコア、不足しているメンテナー用ファイル、テストやドキュメントの不足、優先順位。

## 2. メンテナー向け説明を作る

`brief` は、Codexへ渡す前提情報を1つのMarkdownにまとめたいときに使います。

良い出力: リポジトリの状態、Codex用プロンプト、人間レビューのルール、作業候補。

## 3. 不足点をタスク化する

`tasks` は、準備不足をMarkdown、JSON、GitHub Issue下書きに変えたいときに使います。

良い出力: タスク名、ラベル、検証コマンド、メンテナー確認リスト。

## 4. AI支援の変更をレビューする

ローカルに変更がある状態で `review` を使うと、人間が確認しやすいレビュー資料を作れます。

良い出力: 変更ファイル、リスク、検証コマンド、マージ判断材料。

## 5. GitHub Actions版と組み合わせる

手元で確認したいときはCLI版を使います。定期実行やPRごとの資料作成には [Codex Maintainer Action](https://github.com/goonobu-dot/codex-maintainer-action) を使います。
